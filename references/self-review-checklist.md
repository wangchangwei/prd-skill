# Self-Review Checklist：PRD 与 feature list 完成后的双层自审

> 用于 prd-writer 阶段 3.5（文档自审）。当环境未安装 `superpowers:requesting-code-review` 时，按此清单**自己**走一遍。

## 第一层：内联 spec self-review

> 来自 `superpowers:brainstorming` 的 "Spec Self-Review" 节，4 项扫描。

### 1. Placeholder 扫描
- [ ] PRD.md 中无 `TBD` / `TODO` / `XXX` / `___`
- [ ] feature_list.md 中无空"功能描述"或空"备注"占位
- [ ] 表格里的 `|` 与 `---` 行不存在脱漏

### 2. 内部一致性
- [ ] feature_list 的每一行 `M-XX` 都能在 PRD 详细功能说明节（通常是 "5." 开头那一章）找到对应详细说明
- [ ] PRD 的"非功能要求"指标与"详细功能说明"中的异常处理不冲突
- [ ] mermaid 流程图节点都在功能列表里有出处
- [ ] 数据结构示例字段与 feature 描述一致

### 3. Scope 检查
- [ ] MVP 范围能在 1 个迭代（2-4 周）落地？若不行 → 写出"迭代规划"分阶段
- [ ] 没有"未定优先级"的悬空功能
- [ ] 每个 P0 都是真"上不了线就没法开门营业"的功能（不是 P1 误标 P0）

### 4. 歧义检查
- [ ] 每个"自动" / "智能" / "实时" / "快速"都有可量化定义（如 ≤ 200ms）
- [ ] 每个"用户" / "管理员" 在前文都有明确定义
- [ ] 多义词（如"订单" = 客户端订单 vs 商家工单）只取一个含义并显式声明

## 第二层：换视角 reviewer 自审

> 来自 `superpowers:requesting-code-review` 的精神：换"另一双眼睛"看，分 Critical / Important / Minor。

### 5. 用户旅程闭环
- [ ] 每个核心用户旅程（如注册→下单→支付→收货→退款）首尾相接，无断点
- [ ] 失败路径有兜底（支付失败、网络断开、商品下架）
- [ ] 用户能"撤回"操作（取消订单、删除收藏、注销账号）

### 6. 异常路径覆盖
- [ ] 每个 P0 功能至少有 2 个异常场景 + 处理方式
- [ ] 输入校验规则成文（必填、长度、格式）
- [ ] 并发与幂等：哪些接口要防重复（支付、提交、扣库存）

### 7. 文档可施工性
- [ ] 工程师看 PRD 5.x 节能直接开始拆任务，无需追问"具体什么意思"
- [ ] 所有外部依赖（第三方 SDK、API）都注明文档链接
- [ ] 涉及合规/隐私的数据有显式说明

## 反馈分级与处理

| 等级 | 标准 | 处理 |
|------|------|------|
| **Critical** | 阻塞上线 / 数据丢失 / 合规风险 | 立即修，不进下一阶段 |
| **Important** | 体验严重受损 / 关键路径不闭环 | 进入下一阶段前修 |
| **Minor** | 文案、措辞、可读性 | 记到"已知问题"，可在 V1.1 修 |

## 完成判定

第一层 4 节 + 第二层 3 节，**每节都至少标记一次"已逐条核对"** 才算自审完成。
任何 Critical 未修 → 不允许声明"PRD 完成"。
