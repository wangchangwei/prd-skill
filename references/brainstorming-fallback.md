# Brainstorming Fallback：当 superpowers 不可用时的最小问卷模板

> 用于 prd-writer 阶段 1（需求采集）。当环境未安装 `superpowers:brainstorming` 时，按此清单逐项自行执行 Socratic 澄清。

## 核心原则（来自 brainstorming）

1. **一次一个问题**，不要堆问
2. **多选优先于开放**（用户更容易回答）
3. **YAGNI**：删掉一切非必要功能
4. **提 2-3 个方案带 trade-offs**，再给推荐
5. **每一节设计完询问用户**，得到批准再往下

## 至少要问的 3 个问题

### Q1：核心用户与场景（who + when + why）
> "这个产品主要给谁用？典型使用场景是什么？"
- (a) 选项 A：[根据上下文猜的用户群]
- (b) 选项 B：[另一种可能]
- (c) 其他（请描述）

### Q2：MVP 边界（scope）
> "下面哪些功能必须在 MVP 里？哪些可以放到 V1.1+？"
- 列出从原始描述中识别出的 5-8 个候选功能
- 让用户标 P0 / P1 / P2 / 不要

### Q3：关键约束（technical / business）
> "有没有不能踩的红线？比如必须用某栈、必须接某 SaaS、上线 deadline？"

## 可选问题（按场景追加）

- 用户量级与并发？（→ 影响架构）
- 是否需要管理后台？（→ 影响范围）
- 数据敏感度？（→ 影响合规）
- 是否多端（小程序 / Web / App）？（→ 影响交付）
- 是否需要支付？（→ 影响接入难度）

## 2-3 个方案对比模板

完成提问后，按下面格式列方案：

```
## 方案对比

| 维度 | 方案 A：__ | 方案 B：__ | 方案 C：__（可选） |
|------|----------|----------|------------------|
| 适用场景 | | | |
| 实现成本 | | | |
| 上线时间估计 | | | |
| 关键风险 | | | |

**推荐：方案 X，因为 ___**
```

## 用户审批 gate

提完方案，**必须等用户回复"go"或选定方案** 才进入阶段 2，不要自作主张。

## 转交下游

用户确认后，按 [references/prompts.md](prompts.md) 的格式输出结构化需求，再进入阶段 2 Feature List。
